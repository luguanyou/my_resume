# 项目技术实现 - 面试问答手册

> 本文档针对「个人作品集与交互式简历 Agent」项目，整理高频面试技术问题与真实代码实现细节。
> 技术栈：FastAPI + SQLite + ChromaDB + LangChain + DeepSeek / OpenAI + SSE + Docker

---

## 一、整体架构

### Q：介绍一下这个项目的整体架构？

**A：** 项目分两条数据链路并行运转：

**链路一：结构化数据（REST API）**
```
resume.json → SQLite（SQLAlchemy ORM）→ FastAPI 接口 → 前端展示
```
负责简历的结构化字段展示，如个人信息、工作经历、项目、技能、教育背景，数据稳定、接口标准。

**链路二：非结构化知识（RAG + AI 对话）**
```
resume_knowledge.md → LangChain 文本分块 → ChromaDB 向量库
                                                    ↓
用户提问 → 向量检索 Top-K 片段 → 拼入 System Prompt → DeepSeek LLM → SSE 流式输出
```
负责自然语言问答，AI 扮演"数字分身"回答访客问题。

**启动顺序（`app/main.py`）：**
```python
@app.on_event("startup")
async def on_startup():
    init_database()      # 1. 建表 + 种子数据
    refresh_knowledge()  # 2. 读 .md → 向量化 → 写入 ChromaDB
```
服务启动时自动完成数据库初始化和知识库向量化，无需人工干预。

## 二、为什么选 FastAPI

### Q：为什么用 FastAPI 而不是 Django 或 Flask？

**A：** 三个核心原因：

**1. 原生异步支持**
SSE 流式接口要求服务端持续推送数据，必须用 `async def` 非阻塞 IO。FastAPI 基于 Starlette，天然支持 `asyncio`，而 Django 异步支持较复杂，Flask 需要额外配置。

```python
# endpoints.py - 异步生成器实现 SSE
async def _sse_generator(session_id: str, message: str) -> AsyncGenerator[str, None]:
    async for token in chat_stream(session_id, message):
        payload = json.dumps({"content": token}, ensure_ascii=False)
        yield f"data: {payload}\n\n"
```

**2. Pydantic 自动验证**
请求参数和响应模型用 Pydantic Schema 定义，自动做类型校验，少写大量 if 判断代码，还能自动生成 OpenAPI 文档（`/docs`）。

```python
@router.get("/profile", response_model=UserProfileSchema)
def api_profile(db: Session = Depends(get_db)):
    ...
```

**3. 依赖注入系统**
数据库 Session 通过 `Depends(get_db)` 注入，框架自动管理生命周期（用完关闭连接），避免连接泄漏，代码解耦干净。

## 三、数据库设计

### Q：为什么用 SQLite 而不是 PostgreSQL？

**A：** 这是个人简历展示项目，数据量极小（几十条记录），选型原则是"够用就好，部署最简"：

- **SQLite**：无需独立服务进程，数据是一个 `resume.db` 文件，Docker 部署只需挂载一个 Volume，运维成本为零。
- **PostgreSQL**：适合高并发写入、多用户、数据量大的场景，对本项目来说是过度设计。

如果后续业务量上来，SQLAlchemy ORM 层屏蔽了底层差异，只需改一行配置：
```python
# config.py - 改这一行即可切换数据库
DATABASE_URL: str = "postgresql://user:pass@host/db"
```

### Q：数据库 Session 怎么管理的，会不会泄漏？

**A：** 用 FastAPI 的依赖注入 + Python 生成器模式管理：

```python
# database.py
def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db       # 接口处理期间持有 Session
    finally:
        db.close()     # 无论成功还是报错，finally 保证关闭
```

`get_db` 是一个生成器函数，`yield` 前是获取资源，`yield` 后（finally 块）是释放资源。FastAPI 在请求结束后自动驱动生成器执行 finally，不会泄漏。

### Q：init_db 里为什么要判断 count == 0？

**A：** 防止重复写入。服务每次重启都会执行 `init_database()`，如果不加判断，每次都写一遍种子数据，数据就会重复叠加。

```python
if db.query(UserProfile).count() == 0:
    _seed_data(db, data)
```

**注意**：正因为这个判断，改了 `resume.json` 后，必须先删除 `resume.db`，重启服务才能生效。

## 四、RAG 实现原理

### Q：RAG 是什么，你是怎么实现的？

**A：** RAG（Retrieval-Augmented Generation，检索增强生成）是解决大模型"幻觉"问题的标准方案。核心思想是：**不让模型凭记忆回答，而是先检索相关资料，再让模型基于资料回答**。

我的实现分两个阶段：

**阶段一：离线建库（服务启动时执行）**

```
resume_knowledge.md
    ↓ LangChain RecursiveCharacterTextSplitter 按语义分块（每块 500 字，重叠 80 字）
    ↓ ChromaDB 内置 Embedding 模型向量化
    ↓ 存入 chroma_db/ 持久化目录
```

```python
# document_loader.py - 按语义边界分块，优先在标题/段落处断开
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=80,
    separators=["\n## ", "\n### ", "\n\n", "\n", "。", "；", " "],
)
```

**阶段二：在线检索（每次用户提问时执行）**

```
用户问题 → 向量化 → 在 ChromaDB 中余弦相似度检索 Top-3 片段
    ↓
拼入 System Prompt → 发给 DeepSeek LLM → 流式返回答案
```

```python
# vector_store.py - 余弦相似度检索
_collection = client.get_or_create_collection(
    name=COLLECTION_NAME,
    metadata={"hnsw:space": "cosine"},  # 余弦相似度
)
results = collection.query(query_texts=[query], n_results=top_k)
```

### Q：为什么选 ChromaDB 而不是 FAISS 或 Pinecone？

**A：** 三者适用场景不同：

| | ChromaDB | FAISS | Pinecone |
|---|---|---|---|
| 部署方式 | 本地文件持久化 | 纯内存（重启丢失）| 云服务（需付费）|
| 适用场景 | 个人/小型项目 | 大规模离线检索 | 企业级生产 |
| 运维成本 | 极低 | 需自建持久化 | 零运维但有成本 |

ChromaDB 支持本地文件持久化，配合 Docker Volume 挂载，数据不丢失，完全满足本项目需求，且无需任何外部依赖。

### Q：chunk_overlap（分块重叠）有什么用？

**A：** 防止关键信息被截断在两个块的边界处。比如一句话"我在扬州大学取得了计算机技术硕士学位，期间发表了一篇北大核心期刊论文"，如果刚好在"硕士学位"后面断开，两个块各自都不完整，检索时可能两个都匹配不上。80字的重叠保证了相邻块之间有内容衔接。

## 五、SSE 流式对话

### Q：为什么用 SSE 而不是 WebSocket？

**A：** 两者都能做实时推送，选 SSE 的原因：

**对话场景是单向推送**：用户发一条消息，服务器流式返回，交互模型是"请求-响应"，不需要 WebSocket 的双向全双工通道。SSE 是 HTTP 协议，更轻量，天然支持断线重连、负载均衡代理。

**WebSocket 适合的场景**：实时协作编辑、多人游戏、聊天室（需要双向频繁通信）。

### Q：SSE 的数据格式是什么，前端怎么接收？

**A：** SSE 是纯文本协议，每条消息格式固定：

```
data: {"content": "你好"}\n\n
data: {"content": "，"}\n\n
data: {"content": "我是"}\n\n
data: [DONE]\n\n
```

我的实现（`endpoints.py`）：
```python
async def _sse_generator(session_id: str, message: str):
    async for token in chat_stream(session_id, message):
        payload = json.dumps({"content": token}, ensure_ascii=False)
        yield f"data: {payload}\n\n"   # 固定格式：data: + 内容 + 两个换行
    yield "data: [DONE]\n\n"           # 结束信号
```

响应头需要设置三个关键字段：
```python
headers={
    "Cache-Control": "no-cache",        # 禁止缓存，保证实时性
    "Connection": "keep-alive",          # 保持长连接
    "X-Accel-Buffering": "no",          # 禁止 Nginx 缓冲，否则数据会攒一批再发
}
```

前端用原生 `EventSource` 或 `fetch` + `ReadableStream` 接收。

### Q：多轮对话的上下文是怎么保存的？

**A：** 用内存字典按 `session_id` 存储对话历史：

```python
# chat_engine.py
_session_store: dict[str, list[dict]] = defaultdict(list)
MAX_HISTORY_ROUNDS = 5  # 最多保留最近 5 轮
```

每轮对话追加用户消息和 AI 回复，超过 5 轮自动滑动窗口截断最旧的记录，控制 Token 用量。

**注意**：当前是进程内存存储，服务重启后历史丢失。生产环境可改用 Redis 持久化，改动只在 `_session_store` 那一层。

## 六、System Prompt 设计

### Q：你的 AI 助手 Prompt 是怎么设计的，为什么这么设计？

**A：** Prompt 模板（`chat_engine.py`）：

```python
SYSTEM_PROMPT_TEMPLATE = """你是「卢官有」的数字分身和专属简历助手。
基于以下检索到的简历知识库内容，以专业、真诚且友好的语气回答访客问题。

回答要求：
1. 只基于提供的上下文信息回答，不要编造不存在的内容。
2. 如果上下文中没有相关信息，请诚实地说"这方面的信息我暂时没有"。
3. 用第一人称（"我"）来代表卢官有进行回答。

以下是检索到的相关简历信息：
{context}
"""
```

**设计要点：**

1. **角色定义**：明确告诉模型"你是谁"，防止模型跑题扮演其他角色。
2. **接地原则**：只允许基于检索到的上下文回答，杜绝幻觉，这是 RAG 的核心价值。
3. **兜底话术**：没有信息时有固定措辞，保持体验一致性。
4. **第一人称**：让 AI 以"我"自称，营造"数字分身"的沉浸感。

## 七、CORS 跨域

### Q：CORS 是怎么配置的，为什么要配置？

**A：** 浏览器的同源策略会阻止前端（如 `localhost:3000`）请求不同域名的后端（如 `api.domain.com`），CORS 中间件在响应头里加入许可信息，告诉浏览器"允许跨域"。

```python
# security.py
app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.CORS_ORIGINS,  # 生产环境改为具体域名
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

开发环境 `CORS_ORIGINS=["*"]` 允许所有来源，生产部署时应改为具体域名（如 `["https://luguanyou.com"]`），避免安全风险。

## 八、Docker 部署

### Q：项目是怎么容器化的，为什么要用多阶段构建？

**A：** Dockerfile 采用两阶段构建：

```dockerfile
# 阶段一：builder（安装依赖）
FROM python:3.11-slim AS builder
RUN pip install --prefix=/install -r requirements.txt

# 阶段二：运行镜像（只复制产物）
FROM python:3.11-slim
COPY --from=builder /install /usr/local
COPY app/ ./app/
```

**为什么这么做：**
- 阶段一装了 gcc、编译工具等构建依赖，体积大。
- 阶段二只复制编译好的 Python 包，不包含构建工具，最终镜像体积减少 40%-60%。
- 镜像越小，拉取越快，攻击面越小。

**Volume 挂载持久化数据：**
```yaml
# docker-compose.yml
volumes:
  - ./resume.db:/app/resume.db        # SQLite 数据库
  - ./chroma_db:/app/chroma_db        # 向量数据库
  - ./resume_knowledge.md:/app/resume_knowledge.md
```

数据文件挂载到宿主机，容器重建后数据不丢失。

## 九、知识库刷新机制

### Q：更新了知识库文档后，怎么让 AI 学到新内容？

**A：** 调用管理接口触发重建：

```
POST /api/admin/refresh-knowledge
```

内部执行逻辑（`vector_store.py`）：
```python
def refresh_knowledge() -> int:
    client.delete_collection(COLLECTION_NAME)  # 1. 删除旧向量数据
    _collection = None
    chunks = load_and_split(settings.KNOWLEDGE_FILE)  # 2. 重新读取 .md 文件
    collection.add(documents=chunks, ids=ids)          # 3. 重新向量化写入
    return len(chunks)
```

这个设计让知识库更新无需重启服务，做到热更新。服务启动时也会自动执行一次，保证冷启动后知识库就绪。

## 十、项目亮点总结（用于面试最后陈述）

1. **双库联动架构**：SQLite 存结构化数据，ChromaDB 存非结构化知识，两种数据库各司其职，互为补充，这是 AI 应用的典型架构范式。

2. **真实 RAG 落地**：不是调用 AI 直接回答，而是先检索私有知识库，再交给 LLM 生成，有效防止幻觉，保证回答真实可信。

3. **SSE 流式体验**：逐 Token 实时推送，复刻 ChatGPT 打字机效果，用户体验比等待完整响应好得多。

4. **生产级部署考量**：Docker 多阶段构建控制镜像体积，Volume 持久化数据，环境变量管理敏感配置，CORS 按环境分级配置。

5. **Vibe Coding 实践**：Python / FastAPI / RAG 都是在 AI 辅助下快速上手落地，单人完成从 0 到部署的完整闭环，研发人效提升 200%+。
